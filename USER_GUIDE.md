# Dialog Analytics — Руководство пользователя

## Что делает система

Система автоматически собирает все завершённые диалоги поддержки, анализирует их с помощью искусственного интеллекта и превращает в понятные отчёты. Вместо того чтобы вручную читать сотни переписок, вы получаете готовые выводы: где чаще всего жалуются, кто из операторов работает хорошо, а где есть проблемы.

---

## Откуда берутся диалоги

Система собирает переписки из двух мест:

**JivoChat** — онлайн-чат на сайте. Как только клиент завершает разговор с оператором, диалог автоматически поступает в систему.

**Личные сообщения сайта** — внутренняя переписка между клиентами и организаторами. Система проверяет новые завершённые диалоги каждые 15 минут.

> Диалог считается завершённым, если прошёл 1 час с момента последнего сообщения и оператор его прочитал.

---

## Что анализирует AI

По каждому диалогу искусственный интеллект определяет:

### Кто написал
- **Клиент** — обычный покупатель
- **ПВЗ** — раздающий, пункт выдачи
- **Сотрудник** — внутреннее обращение

### О чём обращение (категория)
Система сама определяет тему из списка:
- Заказ и доставка
- ПВЗ и выдача
- Оплата и финансы
- Технические проблемы
- Аккаунт и профиль
- Качество товара
- Закупки и организаторы
- Пристрой
- Пожелания и инсайты
- Благодарность

### Тип обращения
- **Жалоба** — клиент недоволен, раздражён
- **Проблема** — что-то не работает, без сильного негатива
- **Консультация** — хочет понять как что-то устроено
- **Статус** — где заказ, что происходит
- **Запрос действия** — просит что-то сделать (вернуть деньги, отменить заказ)
- **Благодарность** — позитивное завершение

### Эмоция клиента
- 😊 **Позитив** — выраженная благодарность
- 😐 **Нейтральный** — без выраженных эмоций
- 😠 **Негатив** — жалобы, раздражение, недовольство

### Результат обращения
- ✅ **Решено** — клиент получил конкретный результат
- ❌ **Не решено** — проблема осталась без ответа по существу
- 🔶 **Частично** — дана промежуточная информация, но не закрыто
- 🔺 **Эскалация** — передано другому сотруднику или отделу

> **Важно:** «Решено» ставится только если проблема реально закрыта в диалоге. Если оператор пообещал разобраться или клиент ждёт — это «Частично».

### Оценка оператора (0–100)
- **Отлично (90–100)** — точный, понятный, полезный ответ
- **Хорошо (70–89)** — в целом полезно, но не идеально
- **Удовлетворительно (40–69)** — частичный ответ без чёткого решения
- **Плохо (0–39)** — ответ не помог, нерелевантен

> Оценивается качество ответа, а не результат — иногда проблему объективно нельзя решить быстро, но ответить хорошо — можно всегда.

### Риск оттока (0.0–1.0)
Вероятность что клиент уйдёт или перестанет пользоваться платформой:
- **0.0–0.2** — всё в порядке
- **0.3–0.5** — есть недовольство, не критичное
- **0.6–0.8** — явное раздражение, проблема не решена
- **0.9–1.0** — угрожает уйти, сравнивает с конкурентами

### Требует ли эскалации
Система ставит флаг «требует внимания» если:
- Статус = Эскалация
- Риск оттока ≥ 0.8
- Клиент явно требует компенсацию, возврат или руководство

## Ежедневные отчёты в Telegram

> ⚙️ **Статус:** отчёты формируются вручную. Автоматическая отправка по расписанию будет доступна после настройки cron на сервере (см. раздел «Как запускать вручную»).

После настройки cron отчёт за вчера будет приходить каждый день в 9:00, по понедельникам — недельный отчёт.

### Как читать отчёт

**📊 Итог** — одна строка с главными цифрами:
```
247 диалогов, качество операторов 74/100, решено 61%
↑18 к прошлому периоду
```

**🔴 Топ проблем** — категории с наибольшим объёмом или резким ростом. Под каждой — реальные формулировки из диалогов (не перефразированные):
```
Заказ и доставка — 89 обращений (↑23 к прошлой неделе)
  • "Заказ ещё не отправлен, хотя закупка закрылась 5 дней назад"
  • "Не могу отследить где моя посылка"
  • "Написано что в пути уже 14 дней"
```

**⚠️ Требуют реакции** — только если есть реальные случаи:
- Эскалации с описанием проблемы
- Клиенты с высоким риском оттока
- Операторы с низкой оценкой (имя + сколько диалогов)

**💡 Сигналы** — повторяющиеся паттерны, которые нужно решать системно:
```
Задержка отправки после закрытия закупки — встречается 12 раз
```

> Если в каком-то разделе нет данных — раздел не показывается. Никакой воды и заполнителей.

## SLA — контроль скорости ответа

Система отслеживает насколько быстро операторы отвечают клиентам.

### Правила для личных сообщений (ЛС сайта)

| Когда написал клиент | Дедлайн ответа |
|---|---|
| Пн–Пт с 9:00 до 18:00 | До 18:00 того же дня |
| Пн–Пт после 18:00 | До 11:00 следующего рабочего дня |
| Пятница после 18:00 | До 11:00 понедельника |
| Суббота или воскресенье | До 11:00 понедельника |

### Правила для JivoChat

Jivo работает в рабочее время **понедельник–пятница, 8:00–20:00** (время Самары).
Норматив ответа: **5 минут** с момента первого сообщения клиента.

Если клиент написал вне рабочего времени — дедлайн переносится на начало следующего рабочего дня (8:05).

### Что фиксируется в базе данных

По каждому диалогу система записывает строку в таблицу `dialog_sla`:

| Поле | Что означает | Пример |
|---|---|---|
| `source` | Источник диалога | `jivo` или `site_pm` |
| `chat_id` | ID диалога | `1234062` |
| `client_msg_at` | Когда клиент написал первое сообщение | `2026-04-16 10:15:00` |
| `operator_msg_at` | Когда оператор ответил первый раз | `2026-04-16 10:18:30` |
| `sla_deadline` | Дедлайн по правилам SLA | `2026-04-16 18:00:00` |
| `response_minutes` | Сколько минут до первого ответа | `3` |
| `sla_violated` | Нарушен ли SLA: 1 = да, 0 = нет | `0` |
| `is_open` | Тред ещё открыт и без ответа: 1 = да | `0` |

> Если оператор ещё не ответил — поля `operator_msg_at` и `response_minutes` остаются пустыми.

### Брошенные диалоги

Если оператор не ответил и SLA уже истёк — диалог записывается в ту же таблицу с флагом `is_open = 1` и `sla_violated = 1`.

> ⚙️ **Статус:** данные пишутся в базу. Автоматических уведомлений о нарушениях пока нет — просмотр через SQL-запросы (см. примеры ниже).

## Как запускать вручную

> Для выполнения команд нужен доступ к серверу по SSH.

### Посмотреть что нашлось в ЛС (без обработки)
```bash
cd /opt/jivo_inspector
venv/bin/python3 poller.py --source site_pm --dry-run --limit 10
```

### Запустить обработку диалогов вручную
```bash
cd /opt/jivo_inspector
venv/bin/python3 poller.py --source site_pm --limit 20
```

### Получить отчёт за сегодня прямо сейчас
```bash
cd /opt/jivo_inspector
venv/bin/python3 report.py --period day --dry-run   # показать в консоли
venv/bin/python3 report.py --period day              # отправить в Telegram
```

### Проверить SLA нарушения
```bash
cd /opt/jivo_inspector
venv/bin/python3 sla_checker.py --mode open --dry-run
```

### Проверить работу сервиса
```bash
curl http://localhost:62000/health
```

## Как читать данные в ClickHouse

Если у вас есть доступ к ClickHouse напрямую (например через DBeaver или Grafana):

### Топ проблем за сегодня
```sql
SELECT category, subcategory, count() AS cnt
FROM dialog_analysis
WHERE toDate(analyzed_at) = today()
GROUP BY category, subcategory
ORDER BY cnt DESC
LIMIT 20
```

### Операторы с плохой оценкой за неделю
```sql
SELECT operator_name, round(avg(a.agent_quality_score), 1) AS avg_score, count() AS cnt
FROM dialog_analysis a
JOIN dialogs d ON a.chat_id = d.chat_id
WHERE toDate(a.analyzed_at) >= toMonday(today())
  AND a.agent_quality_label IN ('Плохо', 'Удовлетворительно')
GROUP BY operator_name
ORDER BY avg_score ASC
```

### Сводка по SLA за сегодня
```sql
SELECT
    source,
    count()                                        AS всего,
    countIf(sla_violated = 1)                      AS нарушений,
    countIf(sla_violated = 0)                      AS в_срок,
    round(avg(response_minutes), 1)                AS среднее_время_мин,
    countIf(is_open = 1 AND sla_violated = 1)      AS брошенных_без_ответа
FROM dialog_sla FINAL
WHERE toDate(client_msg_at) = today()
GROUP BY source
```

### Нарушения SLA за сегодня — список диалогов
```sql
SELECT source, chat_id, client_msg_at, sla_deadline, response_minutes
FROM dialog_sla FINAL
WHERE toDate(client_msg_at) = today()
  AND sla_violated = 1
  AND is_open = 0
ORDER BY client_msg_at ASC
```

### Брошенные прямо сейчас (нет ответа + SLA истёк)
```sql
SELECT source, chat_id, client_msg_at, sla_deadline
FROM dialog_sla FINAL
WHERE is_open = 1
  AND sla_violated = 1
ORDER BY sla_deadline ASC
```

### Среднее время ответа по дням за последние 7 дней
```sql
SELECT
    toDate(client_msg_at)      AS день,
    source,
    round(avg(response_minutes), 1) AS среднее_мин,
    countIf(sla_violated = 1)  AS нарушений
FROM dialog_sla FINAL
WHERE client_msg_at >= today() - 7
  AND is_open = 0
GROUP BY день, source
ORDER BY день DESC, source
```

## Часто задаваемые вопросы

**Почему диалог не появился в аналитике?**
- Прошло менее 1 часа с последнего сообщения
- Оператор ещё не прочитал тред (is_unread=1)
- Диалог без текстовых сообщений
- Поллер ещё не успел запуститься (ждите до 15 минут)

**Почему AI поставил категорию неправильно?**
AI иногда ошибается на коротких или неоднозначных диалогах. Если ошибки повторяются системно — сообщите, доработаем промпт.

**Почему отчёт не пришёл в Telegram?**
Проверьте что в `.env` указаны `TELEGRAM_BOT_TOKEN` и `TELEGRAM_CHAT_ID`. Запустите вручную с `--dry-run` чтобы убедиться что данные есть.

**Можно ли перепрогнать анализ по старым диалогам?**
Да. Запустите:
```bash
venv/bin/python3 reprocess.py --source site_pm --limit 100
```

**Что значит «риск оттока 0.85»?**
AI оценил что клиент с высокой вероятностью недоволен настолько, что может прекратить пользоваться платформой. Это сигнал для проактивного контакта или решения его проблемы.
